# VM-Based Sandboxes

## Context

Agent runs execute inside a sandbox that holds the code checkout, the task file, harness
config, and the driver binary. Until now the only sandbox was a Docker container
(`DockerSandboxService`). Containers share the host kernel, which caps the isolation we can
offer and blocks features that need a real kernel â€” most importantly running Docker *inside*
the sandbox, but also future port exposure and snapshotting.

We wanted a second sandbox type that provides hardware-level isolation while reusing the
existing `SandboxService` abstraction so the rest of the system (notably
`WorkflowExecutionService`) does not care which kind of sandbox a run uses.

## Decision

- **VM technology**: Cloud Hypervisor on Linux (KVM) and vfkit on macOS
  (Apple's Virtualization.framework, hardware-accelerated on Apple Silicon). Both support
  **virtio-fs** for hostâ†”guest directory sharing and expose a **REST API over a Unix socket**
  for lifecycle control, and both run as **one process per VM**. Chosen over Firecracker (no
  virtio-fs) and QEMU (slower boot, no REST API).
- **Platform abstraction**: a `VmPlatformAdapter` interface (`CloudHypervisorAdapter`,
  `VfkitAdapter`) hides the per-VMM differences. `VmSandboxService` implements `SandboxService`
  and delegates to whichever adapter matches the host platform.
- **File sharing = virtio-fs**: the run's temp directory is shared into the guest at
  `/mnt/host`. Because the guest writes through to the host filesystem in real time, the
  existing git push/merge flow in `workflow.onTaskCompleted()` is unchanged.
- **Boot chain**: a minimal init (`/sbin/vm-init`, baked into the rootfs) mounts the
  filesystems and virtio-fs, then runs a per-run `vm-mount-setup.sh` (generated by
  `VmSandboxService`) that maps volumes to the paths a container would see (`/code`,
  `/task.txt`, â€¦) and runs `lifecycle.sh` â€” the same lifecycle script the Docker path uses.
- **Rootfs from the Dockerfile**: `pnpm vm:build-rootfs` builds the existing image, exports it,
  and converts it to a raw ext4 disk â€” one source of truth for agent tooling. It also builds a
  small busybox **initramfs** and downloads a prebuilt kernel.
- **Sandbox type config**: a `sandbox_type_configuration` table (project + workspace levels).
  Resolution is project â†’ workspace â†’ default `docker`. Exposed via HTTP
  (`/sandbox-type` on workspaces and projects) and an MCP tool; selectable in the workspace and
  project dialogs in the UI.
- **`WorkflowExecutionService` selects per run**: it receives both sandbox services plus the
  config repository, resolves the type for the project, and uses the matching service. MCP and
  driver host-API URLs are also chosen per type (Docker reaches the host via
  `host.docker.internal`; a VM reaches it via the bridge IP).

`VmSandboxService` generates `vm-mount-setup.sh`, starts the VMM (and virtiofsd on Linux) via
the adapter, monitors the VMM process for exit, and tears everything down on stop.

## Setup

VM env vars are all optional; if they are unset the server runs Docker-only and the VM run path
fails with a clear error only if a run actually requests a VM. Relevant vars: `VM_KERNEL_PATH`,
`VM_ROOTFS_PATH`, `VM_INITRD_PATH`, `VM_HOST_BRIDGE_IP`, the binary paths
(`CLOUD_HYPERVISOR_PATH`, `VIRTIOFSD_PATH` on Linux; `VFKIT_PATH` on macOS), and on Linux the
guest NIC wiring (`VM_TAP_DEVICE`, optionally `VM_MAC`).

`VM_HOST_BRIDGE_IP` is the host IP the in-VM guest uses to reach the driver host-API and MCP server.
It is **platform-specific and has no default** â€” set it to the Linux bridge gateway, or on macOS to
the vmnet/NAT gateway (`192.168.64.1` by default). A VM run without it fails with a clear error.

### macOS (Apple Silicon)

1. `brew install vfkit`.
2. `pnpm vm:build-rootfs` â€” produces `.vm/rootfs.raw`, `.vm/Image-arm64`, and
   `.vm/initramfs.cpio.gz` (requires Docker for the build steps).
3. Set `VFKIT_PATH`, `VM_KERNEL_PATH`, `VM_ROOTFS_PATH`, `VM_INITRD_PATH` to those paths, and
   `VM_HOST_BRIDGE_IP=192.168.64.1` (Virtualization.framework's vmnet NAT gateway).
4. Verify with `pnpm vm:smoke-test` (boots a VM and checks a hostâ†”guest virtio-fs round-trip **and**
   that the guest brought up NAT networking). No host-side bridge setup is needed on macOS â€” vfkit
   uses Virtualization.framework's built-in NAT â€” but the VM still attaches a `virtio-net,nat` NIC
   and the guest DHCPs an address in the initramfs so the in-VM driver can reach the host.

### Linux

1. Install `cloud-hypervisor` and `virtiofsd`; ensure `/dev/kvm` is available. Set
   `CLOUD_HYPERVISOR_PATH` and `VIRTIOFSD_PATH`.
2. `pnpm vm:build-rootfs` (produces an x86_64 `bzImage` kernel + rootfs + initramfs).
3. `sudo bash scripts/setup-vm-networking.sh` once â€” creates the `br0` bridge, enables IP
   forwarding, and sets up NAT so VMs reach the host MCP server and the internet.
4. Set `VM_HOST_BRIDGE_IP` to the bridge gateway so the in-VM driver and MCP client reach the host
   (no default â€” see the Setup note above).
5. Create a TAP device attached to the bridge and set `VM_TAP_DEVICE` (and optionally `VM_MAC`).
   Without it the VM would have no NIC, so `CloudHypervisorAdapter` refuses to start a VM rather
   than letting the run fail opaquely at the workflow timeout.

## Consequences

- A VM run reuses the entire existing run pipeline; only the sandbox implementation differs.
- The kernel boots a busybox initramfs that mounts the rootfs disk and `switch_root`s into
  `/sbin/vm-init`. This is required because Virtualization.framework does not mount the root
  disk itself and vfkit's Linux bootloader needs an initrd.
- A VM has no container-style exit code, so `vm-mount-setup.sh` writes `lifecycle.sh`'s exit
  code to the shared dir and powers the VM off cleanly (a bare `exec` would leave PID 1 to exit
  and panic the kernel). `VmSandboxService` reads that file to decide completed vs error. The
  script never uses `set -e`: as PID 1, any uncaught error would panic the kernel and leave the
  VMM hung, so setup errors are tracked and still fall through to the clean power-off.
- Each VM boots from its own per-run copy-on-write clone of the base rootfs (APFS clonefile on
  macOS, reflink on Linux, via the platform adapter), not the shared base image. This is required:
  the VMM demands exclusive read-write access to a disk image, so attaching one base image to
  concurrent VMs fails â€” and it also isolates per-run disk writes. The clone is reclaimed on stop.
- The macOS/vfkit path is exercised by `pnpm vm:smoke-test`, which also checks NAT networking. The
  Linux/cloud-hypervisor path mirrors the same boot model but has not yet been booted on a
  Linux/KVM host and should be re-verified there.

## Out of scope (future)

Docker-in-VM (the primary long-term motivation), dynamic port exposure, VM snapshotting, and an
in-VM log-streaming daemon. The architecture leaves room for all of these (e.g. the guest serial
console is already captured to a file per run). Concurrent runs are supported via per-run CoW rootfs
clones; sharing a single read-only base across VMs with overlays (to further cut memory) remains a
possible future optimisation.
